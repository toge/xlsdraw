# xlsdraw リポジトリレビュー — 改善点・性能向上案

## 概要

xlsdraw はヘッダオンリの C++20/23/26 ライブラリで、XLSX ファイルに図形（シェイプ・テキストボックス・コネクタ）を描画する OpenXML 生成機能を提供する。レイヤ構成（`units` → `drawing` → `resource`/`io` → `workbook`）はクリーンで、`std::expected`/`std::span`/`fmt` を活用したモダンな C++23 実装。Doxygen コメントも行き届いている。

ただし、**機能していない機能・潜在的な破損バグ・ヘッダ衛生問題・ビルドスクリプトの欠陥**が多数見つかった。以下に優先度別に整理する。

---

## P0 — 正確性バグ（即時修正推奨）

### 1. `Shape::anchor` が無視され `editAs` がハードコード
- **場所**: `include/xlsdraw/drawing.hpp:983,1017`
- `generate_shape_anchor`/`generate_connector_anchor` は `shape.anchor` を参照せず `editAs="oneCell"` 固定。`AnchorType` 列挙（`drawing.hpp:56-60`）と `Shape::anchor` フィールド（`drawing.hpp:795`）がデッド。
- **修正**: `AnchorType` → `editAs` のマッピング（`MoveAndSize→"twoCell"`, `MoveButNoSize→"oneCell"`, `NoMoveNoSize→"absolute"`）を追加し `fmt::format` に挿入。

### 2. XML エスケープが一切ない
- **場所**: `drawing.hpp:727`（`TextRun::text`）、`resource.hpp:54`（`Relationship` 各フィールド）、`workbook.hpp:77`（`sheet_name`）、`resource.hpp:111,115`（`part_name`）
- ユーザ入力に `<`/`>`/`&`/`"` が含まれると malformed XML が生成される。
- **修正**: `xml_escape(std::string_view)->std::string` ヘルパを新設し、全挿入点に適用。

### 3. `ArchiveWriter` の暗黙コピー → 二重 `zip_close`
- **場所**: `include/xlsdraw/io.hpp:36`
- ユーザ宣言デストラクタはあるが copy/move 操作は未宣言。copy ctor は暗黙生成され `zip_t*` がポインタコピーされ、2つの dtor で `zip_close` が二重実行される（use-after-free）。
- **修正**: `ArchiveWriter(ArchiveWriter const&) = delete;` および `operator=(...) = delete;` を追加。move は `za_` 所有権移譲＋送り元 null 化で実装するか delete。

### 4. `save()` が `zip_close` 失敗を検知できない
- **場所**: `io.hpp:50-54`（dtor の `zip_close`）、`workbook.hpp:205,224`
- `ArchiveWriter` はローカル変数で、`save()` が `{}` を返した *後* に dtor が `zip_close` を呼ぶ。`zip_close` が失敗しても `save()` は既に成功を返済み → 破損ファイル + 成功結果。
- **修正**: `ArchiveWriter::close()->std::expected<void, WriteError>` を追加（`zip_close` の戻り値を検査）。`save()` は `writer.close()` を明示呼び出しして結果を伝播。

### 5. `zip_open` 失敗が検知不能
- **場所**: `io.hpp:42-45`
- コンストラクタで `za_` が null になっても呼び出し側は即時検知不可。
- **修正**: ファクトリ `ArchiveWriter::open(path)->std::expected<ArchiveWriter,WriteError>` 化、または `is_open()`/`explicit operator bool()` 追加。

### 6. `WorksheetDrawingContext::add_simple_rect` が負の寸法を黙って 0 化
- **場所**: `include/xlsdraw/worksheet.hpp:46-47`
- `pixels_to_emu(...).value_or(0)` で `UnitError::NegativeValue` を握りつぶし、0 サイズの図形が成功する。
- **修正**: `std::expected<..., DrawingManager::Error>` を返しエラーを伝播。

### 7. `rId10` が `rId2` より前にソートされる
- **場所**: `include/xlsdraw/resource.hpp:62`（`std::map`）
- `std::map` は辞書式ソートのため rId10 が rId2 の前に来る。10+ 関係で順序が壊れる。
- **修正**: `std::vector<Relationship>`（挿入順保持）に変更、または ID をゼロ埋め。

## P1 — デッドコード・誤解を招く API

### 8. 未使用・未実装の宣言を削除
- `struct Image`（`drawing.hpp:466-474`）—参照ゼロ。実装するか削除。
- `Shape::color_argb`（`drawing.hpp:793`）・`Shape::text`（`drawing.hpp:794`）—レガシーフィールド、ジェネレータ未参照。フォールバック接続 or 削除。
- `DrawingManager::Error::DuplicateId`（`drawing.hpp:1070`）—返却されない。到達不能 switch 分岐あり。
- `UnitError::InvalidDpi`（`units.hpp:20`）—生成されない（ctor が黙ってデフォルト DPI に置換）。
- `#include <span>`（`io.hpp:5`）—未使用。
- `DrawingGenerator` インスタンス（`drawing.hpp:1108` で不要なオブジェクト生成）—`static`/free 関数化。

### 9. `next_id_` のインクリメント順序
- `drawing.hpp:1088` — 検証前に `next_id_++` するため、失敗時にも ID が進む。検証を先に。

### 10. `std::move` 後の `shape.id` 参照
- `drawing.hpp:1098-1099` — `push_back(std::move(shape))` 後に `shape.id` を読む。現在は trivial なので安全だが脆弱。移動前に `auto const id = shape.id;`。

## P1 — ヘッダ衛生・API 設計

### 11. 予約識別子の include guard
- 全 6 ヘッダで `__XLSDRAW_*_HPP__`（二重下線は予約）。`#pragma once` に変更推奨。
- `units.hpp` は `<cstdint>` を 2 重 include（4,7 行目）。

### 12. namespace スコープの `using namespace`
- `drawing.hpp:906` — `using namespace std::string_view_literals;` が `xlsdraw::drawing` 全体を汚染。関数スコープへ移動。
- `worksheet.hpp:14-15` — `using namespace drawing; using namespace units;` が `xlsdraw::worksheet` を汚染。修飾名に置換。

### 13. `[[nodiscard]]` の不整合
- 欠落: 全 `generate_xml()`/`to_xml()` メンバ（drawing.hpp:40,491,508,544,566,584,623,656,678,718,742,763）、`DrawingGenerator::generate`、`RelationshipManager::add_relationship`/`generate_xml`（resource.hpp:40,50）、`ArchiveWriter::write_*`（io.hpp:62,72,109,118）、`SingleSheetDrawingWorkbookBuilder::add_shape/save/get_shape_mut`（workbook.hpp:117,148,156）、`WorksheetDrawingContext::add_simple_rect/final_xml`（worksheet.hpp:41,66）。一部は付与済みなので一貫化が必要。

### 14. `std::string_view` メンバのダングリングリスク
- `Paragraph::alignment`（`drawing.hpp:502`）・`LineProperties::cap_type`（`drawing.hpp:736`）が `string_view`。一時オブジェクト由来だとダングリング。`std::string` 化 or 非所有ドキュメント化。

### 15. `get_shape_mut` のポインタ無効化ルール未文書化
- `drawing.hpp:1126` — `std::vector` 再確保でポインタ無効化されるが未記載。

### 16. `zip_file_add` への `string_view::data()` 渡し
- `io.hpp:94` — `zip_file_add` は null 終端前提。現在の呼び出し側はリテラルなので安全だが、API としては unsafe。`std::string` + `.c_str()` または前置条件の明記。

### 17. エラー型の不統一
- `units::UnitError`（enum）、`DrawingManager::Error`（enum）、`io::WriteError`（構造体）、`workbook::save()`（`std::expected<void,std::string>`）。workbook 層が構造化エラーを文字列に平坦化し、呼び出し側のプログラム的処理を阻害。`workbook::Error` バリアント導入を検討。

## P1 — 性能改善

### 18. O(N²) の文字列結合
- 全 `generate_xml` で `xml += fmt::format(...)` を反復（`drawing.hpp:919-924`、`resource.hpp:51-58,105-119` 等）。`reserve()` なし。大規模図形で O(N²) コピー。
- **対策**: `std::string::reserve` で事前確保、または `fmt::join`/`fmt::memory_buffer` 使用。

### 19. プリセット図形の線形探索 ×3
- `drawing.hpp:417` の `preset_shape_info` が 133 エントリを `std::find_if` で線形走査。`make_preset_shape` は prst/default_name/render_kind の 3 回走査。
- **対策**: `PresetShape` は密な enum なので `(size_t)preset` インデックスの `constexpr` 配列で O(1) 参照。

### 20. `std::map` を `std::vector` へ
- `resource.hpp:62,135,136` — 小 N ではノードベースの `std::map` より flat vector が高速かつ rId10 バグも解消。

### 21. `generate_style_xml` の定数文字列再構築
- `drawing.hpp:940-949` — 不変文字列を毎回 `std::string` で構築。`static constexpr std::string_view` 化。

### 22. `Marker::to_xml`/`Color::generate_xml` 等の小文字列アロケーション
- 個別 `fmt::format` で毎回ヒープ確保。ホットパスでは予約済みバッファへの蓄積が有利。

## P1/P2 — ビルドシステム・設定ファイル

### 23. `.gitignore` がビルドスクリプト・VSCode 設定を除外 ★重要
- `.gitignore:43,52,53` — `build*.sh`/`test.sh`/`.vscode/` が git 管理外。なのに `README.md:39,40,47` が `build_win64.sh`/`build_mingw.sh`/`./test.sh` の実行を指示。クローン後、新規貢献者は使えない。
- **修正**: スクリプトを git 管理に戻すか、README を `cmake`/`ctest` 直コマンドに書き換え。CMakePresets.json を導入してスクリプトを廃止するのが最善。

### 24. `vcpkg.json` に `builtin-baseline` なし
- 再現性なし。CI キャッシュキーが効かない。`catch2` がテスト専用なのに無条件依存。
- **修正**: `builtin-baseline` 追加、`"features":{"test":{"dependencies":["catch2"]}}` で条件化。

### 25. `CMakeLists.txt` のフラグ問題
- `CMakeLists.txt:27,28,41,44` — `-g3`/`-O3`/`-march=native`/`/Ox` が `CMAKE_CXX_FLAGS` に無条件追加されビルドタイプ無視。`-march=native` は配布ライブラリで非可搬。
- **修正**: `target_compile_options` + `generator-expressions` でビルドタイプ別制御、`-march=native` は option 化または削除。
- 27/28 行目の `-g3` は Release でも入る。41 行目の `-O3` は Release と二重。
- `option()` 宣言が末尾（131-132 行目）にあり、副作用後に定義。先頭へ移動。
- 22 行目の FATAL_ERROR メッセージ「no C++20 support」が C++26/23 優先ロジックと不一致。

### 26. `test/CMakeLists.txt` の `file(GLOB)` に `CONFIGURE_DEPENDS` なし
- `test/CMakeLists.txt:3` — 新規テスト追加時に再 configure されない。`CONFIGURE_DEPENDS` 追加または明示リスト化。

### 27. ビルドスクリプト群の欠陥（git 管理外だが修正推奨）
- `build.sh:10` — `[ == ]` は bashism（shebang は `#!/bin/sh`）。`=` 使用。
- `build.sh:14` — `-Ofast` は `-ffast-math` 含む。ライブラリでは `-O3`。
- `build.sh:30` — `~/vm/vcpkg` 個人パス固定。
- `build.sh:33` — aarch64 検出するのに triplet は常に `x64-linux-static`。
- `build.sh:43` — `--parallel 4` 固定。`$(nproc)` または `CMAKE_BUILD_PARALLEL_LEVEL`。
- `build_mingw.sh:15,16` — `CMAKE_C_COMPILE`/`CMAKE_CXX_COMPILE` は誤（正しくは `_COMPILER`）。
- `build_mingw.sh:27` — `-DSTATIC=ON -DSTRIP=ON` が CMake 側で未定義のデッドオプション。
- `build_win64.sh:13` — `CONAN_LOGGING_LEVEL==10`（`=` 二重）。
- `build_win64.sh:15` — Conan v1 構文（`-if`/`-pr`）。他スクリプトは v2。
- `build_win64.sh:22` — `export C=` は `CC=` の誤り。
- `test.sh` — `cd build` の失敗ハンドリングなし。`ctest --test-dir build -V` が安全。

### 28. `.editorconfig` の無効値
- `.editorconfig:11` — `indent_size = space` は無効（整数必須）。`indent_size = 2` に修正。
- `*.sh`/`CMakeLists.txt`/`*.json`/`*.yml`/`*.md` にルールなし。

### 29. `.clang-tidy` の設定ミス
- `.clang-tidy:21` — `portabilty-*` はスペルミス（`portability-*`）。portability チェックが実効していない。
- `.clang-tidy:25` — `ClassCase: lower_case` が実際の PascalCase コードと矛盾。lint が未実行の証拠。
- `bugprone-exception-escape` が 2 重記載。
- clang-format 鍵が非推奨（`AlignConsecutiveDeclarations: true` → `Consecutive` 等）。
- `.clang-format:44` — `Standard: Cpp11` が C++20/26 と不一致。`Latest` に。

### 30. CI のギャップ
- `.github/workflows/ci.yml` — Linux ジョブが `CMAKE_BUILD_TYPE` 未設定（最適化なしビルド）。
- clang-format/clang-tidy/sanitizer ジョブなし。JUnit 結果がアーティファクト未アップロード。

## 検証方法

- `cmake -B build -DCMAKE_BUILD_TYPE=Release` && `cmake --build build` でビルド成功。
- `ctest --test-dir build -V` で全テスト緑。
- 新規テスト: anchor 動作・XML エスケープ・`ArchiveWriter` コピー削除・`save()` の `zip_close` 失敗・負寸法エラー・rId 順序。
- `clang-format --dry-run -Werror` と `clang-tidy` の CI 追加で lint 強制。
- example 生成 `.xlsx` を Excel/LibreOffice で開いて描画確認（特に anchor 動作）。
